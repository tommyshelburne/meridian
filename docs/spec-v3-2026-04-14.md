## 1. Vision & Strategic Context

### 1.1 What Meridian Is

Meridian is a closed-loop government BD intelligence and outreach platform. It ingests procurement opportunities from federal and all 50 state sources, scores them against KomBea's lane, identifies real human POCs, drafts and sends outreach, sequences follow-ups, and tracks every interaction through to meeting booked or deal closed — all autonomously.

BidMatch v2 was a discovery tool. Meridian is a revenue engine.

### 1.2 Why v3

BidMatch v2 proved the core loop (ingest → score → CRM → email) works. The gaps it revealed:
- **Outreach ROI is low.** Emails go out but follow-up is manual or absent.
- **POC data is shallow.** Agency names, not people.
- **Sources are narrow.** SAM.gov + one Utah scraper. 49 states and dozens of cooperative vehicles are invisible.
- **Scoring is static.** Fixed keywords, no learning, no seat-count awareness.
- **No audit trail.** No way to know what was sent, to whom, when, with what result.
- **Email infrastructure is fragile.** MS Graph under personal auth caused a calendar loop bug, P1 concurrency issues, and mistaken replies.
- **No portal.** Everything is blind — no visibility into pipeline without opening Pipedrive manually.

### 1.3 Strategic Value

Meridian is equity-bearing IP. It is being built as a licensable SaaS product from day one, even while initially operated as KomBea's internal BD system. The incentive structure is direct: 500 licenses = $500 bonus, 1000 licenses = $1000 bonus. High-seat contracts (prime teaming agreements, state enterprise licenses) are the path to that target.

This means every architectural decision must support multi-tenancy, white-labeling, and eventual productization — even if v3.0 runs a single tenant.

### 1.4 Relationship to Memento

Memento (mnto.dev) is Tommy's Mac-local Claude Code orchestration harness. It provides persistent multi-session RAG, self-written architectural rules, and governance enforcement during build sessions. Meridian will be built *with* Memento — all Claude Code sessions on the Mac should run under Memento so it can accumulate architectural context across the full build.

Memento is a **design-time intelligence layer**, not a runtime dependency. It does not run on Burrow and is not called at runtime. Runtime intelligence (outreach memory, tone adaptation, contact history) is handled by pgvector on Burrow.

## 3. Architecture

### 3.1 Guiding Principles

1. **Clean Architecture is retained.** Domain → Application → Infrastructure → Worker. Dependencies flow inward. No infrastructure concerns in Domain or Application layers.
2. **Multi-tenancy from day one.** Every entity that varies per customer is tenant-scoped. Schema uses `TenantId` discriminator throughout. v3.0 runs one tenant (KomBea). v3.1 adds tenant management.
3. **Ports are sacred.** New capabilities are added as new port interfaces + new adapters. The pipeline orchestrator does not know about concrete implementations.
4. **Immutable audit log.** All state-changing operations append to an audit log. Nothing is deleted from the log.
5. **Explicit failure over silent degradation.** `ServiceResult<T>` propagates through all cross-boundary calls. Exceptions are caught at the worker boundary and logged. No swallowed errors.
6. **Config-driven, not code-driven.** Lane keywords, scoring weights, sequence templates, send windows — all in config/database, not hardcoded. This is what makes Meridian licensable.

#### Scoring Model (v2)

The v2 scoring engine extends v1's 5 dimensions with 3 new signals:

| Dimension | Max Points | v2 Changes |
|---|---|---|
| Lane fit (title) | 2 | Unchanged |
| Lane fit (description) | 1 | Unchanged |
| Agency tier | 2 | Tier table expanded; prime contractor BD contacts = tier 1 |
| Win themes | 2 | Now includes recompete signal (+1 if incumbent is a Nuance/legacy platform) |
| Past performance | 2 | Unchanged |
| Procurement vehicle | 2 | Cooperative vehicle bonus added |
| **Seat count signal** | **2** | **NEW: estimated seats ≥ 100 = 2pts, ≥ 50 = 1pt, unknown = 0** |
| **Recompete bonus** | **1** | **NEW: +1 if detected recompete of a known competitor contract** |

Max score: 14 (up from 10). Thresholds scale proportionally:
- **Pursue**: ≥ 10
- **Partner**: ≥ 6
- **NoBid**: < 6

#### Seat Count Estimation

Estimated seats are derived from contract value signals:
- Explicit seat count in title/description (regex extraction)
- Contract value / $199 per agent per month / 12 months (KomBea pricing model)
- Agency size proxies (tier 1 federal = assume ≥ 200 agents unless scoped otherwise)

This is a heuristic, not a guarantee. Stored as `EstimatedSeats` with a `SeatEstimateConfidence` enum (High/Medium/Low/Unknown).

### 3.6 Email Infrastructure — Graph Service Account

**Problem with v2:** Personal OAuth tokens expire mid-run, calendar permissions caused the loop bug, personal mailbox threading created confusion.

**v3 solution:** App-only authentication via Microsoft Entra ID client credentials flow.

**Setup (one-time, requires M365 admin — Dave or Art):**

1. Register app in Microsoft Entra ID: `Meridian Outreach`
2. API permissions: `Mail.Send` (Application permission) only. No Calendars, no Mail.Read, no Users.
3. Exchange restriction via PowerShell:
   ```powershell
   New-ApplicationAccessPolicy -AppId <MeridianAppId> \
     -PolicyScopeGroupId outreach@kombea.com \
     -AccessRight RestrictAccess \
     -Description "Meridian: restrict to outreach mailbox only"
   ```
4. Store in secrets: `MeridianGraph:TenantId`, `MeridianGraph:ClientId`, `MeridianGraph:ClientSecret`
5. Client secret rotation scheduled annually (calendar reminder in KomBea calendar, not system calendar)

**Result:** BidMatch/Meridian can only send from `outreach@kombea.com`. Calendar access is structurally impossible. Token is stable (no user session, no expiry mid-run). Rotation is scheduled, not reactive.

**Deliverability setup (alongside Entra registration):**
- SPF record: include `outlook.com` (M365 already handles this for @kombea.com)
- DKIM: enable in M365 admin for kombea.com if not already on
- DMARC: verify `p=quarantine` or stronger is set
- Send warm-up: start at 20 emails/day, increase by 10/day over 2 weeks

### 3.10 Sequence Engine Design

The sequence engine is the core of the "close the loop" capability. Design rules:

1. **Enrollment is explicit.** A human or the pipeline orchestrator must enroll a contact in a sequence. No auto-enrollment without a scored Pursue/Partner decision.
2. **One enrollment per contact per opportunity.** Duplicate guard at the repository layer.
3. **Steps are conditional.** A step does not send if: contact has replied, contact has bounced, contact has unsubscribed, enrollment is paused, send window has not opened, daily cap is exceeded.
4. **Reply detection stops the sequence.** When `GraphReplyMonitor` matches a reply to an active enrollment, the enrollment transitions to `Replied` and no further steps are sent automatically. The deal in Pipedrive is updated and a portal notification is generated.
5. **Pause/resume is human-controlled.** Portal users can pause any enrollment. Useful when a phone call is in progress.
6. **Templates use Liquid.** All subject lines and bodies are Liquid templates. No string concatenation in code. Templates are stored in the database (editable via portal) and versioned.

### 3.11 POC Search Strategy

Automated enrichment sources (in priority order):

1. **SAM.gov award contacts** — existing `SamGovPocLookup.cs`, extended to return structured `Contact` with confidence scoring
2. **USASpending.gov** — new `UsaSpendingPocEnricher`: query `/api/v2/awards/` for recent awards to the same agency in same NAICS, extract `recipient_primary_business_representative`
3. **Agency directory lookups** — static + dynamic lookup against known agency directory URLs (e.g. `agency.gov/about/leadership`) — HTML scraping, lower confidence
4. **Manual enrichment queue** — when automated sources return nothing, opportunity is queued in portal for human lookup (LinkedIn, phone directories, etc.)

**Contact confidence scoring:**

| Source | Base Confidence |
|---|---|
| SAM.gov (award contact, same contract) | 0.90 |
| SAM.gov (award contact, same agency+NAICS) | 0.70 |
| USASpending.gov | 0.65 |
| Agency directory scrape | 0.50 |
| Manual entry (portal) | 1.00 |

Contacts with confidence < 0.50 are not enrolled in sequences until verified or manually approved.

## 5. Compliance & Legal

### 5.1 CAN-SPAM

CAN-SPAM applies to **commercial email**. Government agency contacts (`.gov` email addresses) are generally not subject to CAN-SPAM, but prime contractor BD contacts at commercial domains are.

**Required for all outreach regardless of contact type:**
- Physical mailing address in email footer (KomBea address)
- Unsubscribe link in every email (one-click, processes within 10 business days per CAN-SPAM)
- No deceptive subject lines
- `From` display name clearly identifies KomBea

**Suppression list:**
- Unsubscribes are permanent per tenant. Contact marked `IsOptedOut = true`. Never overridden by re-enrollment.
- Bounces (hard bounce) mark contact `IsBounced = true`. Permanently suppressed.
- Soft bounces (mailbox full, etc.) retry up to 3 times then suppress.

### 5.2 Gov Outreach Ethics

- Meridian does not impersonate agency personnel
- All outreach is from `outreach@kombea.com`, clearly from KomBea
- No false urgency, no misleading claims about existing relationships
- Sequences stop immediately on opt-out or reply

### 5.3 Data Handling

- No PII is stored beyond what is necessary for outreach (name, title, agency, email, phone)
- Contacts from automated enrichment are not sold or shared
- Per-tenant data isolation enforced at application layer and auditable at database layer
- Secrets never in code or version control — `dotnet user-secrets` or environment variables only

## 6. Incentive Alignment — High-Seat Contract Targeting

Tommy's bonus incentive: $500 for 500 licenses, $1000 for 1000 licenses. KomBea pricing: $199/agent/month.

500 licenses = one large state enterprise deal, or 2-3 prime teaming agreements with substantial deployments.

**Meridian's scoring and outreach should be explicitly tuned for this:**

- **Scoring bonus** for estimated seats ≥ 100 (as defined in §3.3)
- **Separate sequence type** for prime contractor BD contacts — these are the highest-leverage targets for reaching 500+ seats via a single teaming agreement
- **Prime contractor target list** (hardcoded initially, configurable in settings):
  - ATI Government Solutions (IRS $25M call center mod)

## 11. Spec Addendum — Gap Resolutions (2026-04-14)

Gaps identified during spec review. Resolutions approved by Tommy.

### 11.1 Portal vs Worker Deployment

Portal and Worker are **separate processes** with separate systemd units (`meridian-worker.service` and `meridian-portal.service`). They share the PostgreSQL database but run independently. Worker crashing does not take down the dashboard.

### 11.2 Contact Deduplication

Contacts are deduplicated across enrichment sources:
- **Primary match:** email address (case-insensitive). If a contact with the same email already exists, merge — higher-confidence source wins on field conflicts.
- **Secondary signal:** fuzzy match on name + agency when email is absent. Flagged for human review in the portal rather than auto-merged.

Dedup is enforced at the repository layer. No duplicate enrollments for the same human on the same opportunity.

### 11.3 Template Versioning — Snapshot on Enrollment

When a contact is enrolled in a sequence, the template versions for all steps are **frozen at enrollment time**. Edits to templates in the portal only affect future enrollments. Active enrollments continue using the snapshot they started with.

Implementation: `OutreachEnrollment` stores a `SequenceSnapshotId` referencing an immutable copy of the sequence + steps + template bodies at enrollment time.

### 11.4 Tenant Context in the Worker

Worker resolves tenant context by iterating all active tenants:

```csharp
foreach (var tenant in await tenantRepository.GetActiveTenantsAsync(ct))
{
    using var scope = scopeFactory.CreateScope();
    scope.ServiceProvider.GetRequiredService<ITenantContext>().SetTenant(tenant.TenantId);
    await job.ExecuteAsync(scope.ServiceProvider, ct);
}
```

v3.0 has one tenant (KomBea). Pattern is correct from day one for v3.1 multi-tenancy.

### 11.5 Sequence Engine Edge Cases

- **Daily cap hit mid-batch:** Remaining enrollments roll to the **next day's send window**. They are not retried within the same day.
- **Pause/resume:** Delay timer resets from the resume timestamp. If step 2 was due 3 days after step 1 and the enrollment was paused after step 1, step 2 becomes due 3 days after resume — not 3 days after original step 1.
- **Multiple contacts per opportunity:** Each contact is enrolled independently. No batch coupling.

### 11.6 Reply and Bounce Detection

- **Reply matching:** Primary match by `MessageId` / `In-Reply-To` header from Graph. Fallback: normalized subject-line match (`Re:` stripped, case-insensitive) against active enrollments for the same contact.
- **Bounce detection:** Hard bounces detected via NDR messages in the `outreach@kombea.com` inbox (same polling as reply detection). Soft bounces detected via Graph send failure responses.
- **Soft bounce retry:** 3 retries, each delayed by 24 hours. After 3 failures, contact marked `IsBounced = true` and permanently suppressed.

### 11.7 Authorization Model

- **v3.0:** All three users (Tommy, Angel, Art) have full access. No role-based restrictions.
- **v3.1:** Tenant-admin vs. super-admin roles. Tenant-admin can manage their tenant's config, contacts, sequences. Super-admin can manage tenants, global config, system health.

### 11.8 Configuration Storage Rule

- **Portal-editable** (stored in database): scoring config, lane keywords, send limits, suppression list, sequences, templates, per-source on/off toggles.
- **Operator-managed** (stored in `appsettings.json` + environment variables): connection strings, API keys, worker job cadences, Entra auth config, logging levels.

### 11.9 Cooperative Vehicle Scope Reduction

v3.0 MVP scope reduced to **GSA eBuy only**. NASPO, NCPA, and Sourcewell deferred to v3.1. Rationale: each cooperative vehicle is a distinct system requiring its own adapter. One well-tested adapter is better than four incomplete ones in a 3-week build.

### 11.10 Data Growth Management

- **Audit log:** Partitioned by month (`audit_events_2026_04`, etc.) using PostgreSQL native partitioning. Query performance maintained via partition pruning.
- **Opportunities:** Opportunities older than 12 months with status `NoBid` or `Closed` are archived to a `opportunities_archive` table. Active/watched opportunities retained indefinitely.
- **RAG memories:** No automatic pruning. Manual cleanup via portal Memory Browser.

### 11.11 Minor Resolutions

- **Blazor circuit disconnect:** Reconnection overlay with auto-retry (Blazor's built-in `<ReconnectModal>`). Customized to match portal styling.
- **System health monitoring:** Worker heartbeat written to database every 5 minutes. Portal Settings → Integrations page shows last heartbeat for each job. Alert surfaced on Dashboard if any job heartbeat is > 15 minutes stale.
- **Deployment pipeline:** Git push to `main` on Burrow. Rex pulls and runs `dotnet publish` + systemd restart. No CI/CD pipeline in v3.0 — manual deploy via Rex is sufficient for a 3-person team.
- **Testing strategy:** Infrastructure adapter tests use recorded HTTP responses (`HttpMessageHandler` fakes, same pattern as BidMatch v2). Sequence engine tests use a fakeable `ITimeProvider` for clock control. Email sending tests use a `DryRunEmailSender` adapter. State portal scrapers tested against saved HTML snapshots.
- **Blazor Server confirmed** as the portal framework per data-driven analysis (real-time SignalR, no full-page reloads, component model for drawers/modals, 3-user scale makes WebSocket tradeoff irrelevant).
