# AI Help Chat — v3.1 design

**Status:** scoped, not yet built. Captured 2026-04-28 alongside v3.0 help system shipping.

## Why

The v3.0 help system (callouts + `/help` FAQ) covers static "how do I do X" questions. The next-tier value comes from **diagnostic Q&A** — operators asking questions that need both product knowledge AND visibility into their tenant's current state:

- "Why hasn't my sequence sent today?"
- "Which contacts bounced this week?"
- "Show me opportunities scored Pursue but missing a contact."
- "What's blocking the StateLocal sequence from picking up the Utah opp?"

A static FAQ can't answer those. An LLM with grounding on (a) the product spec/docs and (b) the tenant's own data can.

## Surface

A chat panel accessible from any page (probably a slide-out from the right edge, or an `/assistant` route). Single conversation per user session; persisted across sessions for audit.

```
┌─────────────────────────────────┬──────────────────────┐
│                                 │  Meridian Assistant  │
│  (current page)                 │  ─────────────────── │
│                                 │  > Why hasn't the    │
│                                 │    Utah opp sent?    │
│                                 │                      │
│                                 │  Looking at it…      │
│                                 │  - Opp DEMO-...      │
│                                 │    is Status=Scored  │
│                                 │    but has 0         │
│                                 │    contacts. The     │
│                                 │    enricher returned │
│                                 │    empty for...      │
│                                 │  ───────────────     │
│                                 │  [type a question…]  │
└─────────────────────────────────┴──────────────────────┘
```

## Stack

- **Anthropic SDK** (`anthropic` package). Default model `claude-sonnet-4-6` per spec §7.4 cost estimate; fall back to Haiku for low-stakes troubleshooting once we have telemetry on actual usage.
- **Streaming** responses (Server-Sent Events from a minimal API endpoint, consumed by JS in the chat panel). Blazor Server's interactive mode is not great for streaming; a thin JS island works better.
- **Prompt caching** on the system prompt + grounding context blocks. Articles + spec change rarely; per-request cost should be near-zero after the first hit per cache TTL.

## Grounding

Two retrieval sources, blended into the system prompt:

### 1. Static knowledge (RAG over product docs)

- `docs/spec-v3-2026-04-14.md`
- `docs/spec-product-v1-2026-04-18.md`
- `docs/portal-spec-v1-2026-04-19.md`
- `docs/soft-launch-runbook.md`
- `src/Meridian.Portal/Help/Articles/*.md`

Embed these into pgvector at deploy time (`IRagStore` is already scaffolded). Re-index on doc commit via a CI hook.

### 2. Live tenant context (tool use)

A small set of tool functions Claude can call:

| Tool | Purpose |
|---|---|
| `lookup_opportunity(externalId or title)` | Returns status, score, contacts, enrollments, last audit events |
| `list_blocked_enrollments(tenantId)` | Enrollments paused or stuck-due |
| `recent_audit_events(tenantId, since)` | Last N events filtered by type |
| `tenant_config_summary(tenantId)` | OutboundConfig, sources enabled, sequences defined |

Each tool runs scoped to the user's tenant via `ITenantContext` — Claude cannot escape the tenant boundary even if it tries.

## Guardrails

1. **Per-tenant rate limit + spend cap.** Hard ceiling per tenant per day. Log usage per tenant; alert at 80% of cap. Cap is configurable via `Settings → AI Assistant`.
2. **Audit log.** Every prompt + completion stored in `audit_events` with EventType `AssistantQuery`. Enables compliance review and post-hoc training-data curation.
3. **No data exfiltration.** System prompt explicitly forbids posting tenant data to external services or tools other than the registered ones. Tools have an allowlist.
4. **"I don't know" by default.** System prompt instructs the model to say so rather than hallucinate when the grounded context doesn't cover a question. Test this with a held-out eval set.
5. **No write tools in v3.1.** Read-only diagnostics only. v3.2 could add "draft a sequence for me" or "pause this enrollment" with explicit confirmation.

## Cost model

- **Spec §7.4 baseline:** $2-5/day during active outreach for proposal generation. Chat assistant adds incremental cost on top.
- **Estimate per query:** ~3K tokens system + grounding (cached after first hit), ~500-1500 tokens of tool results, ~300 tokens response. Effective cost per query under $0.005 with caching, ~$0.02 without.
- **Per-tenant cap:** default 200 queries/day. At cached rates, ~$1/tenant/day worst case. Operators heavy on troubleshooting will be the outliers.

## Build phases

| Phase | Scope | Estimate |
|---|---|---|
| 1 | Embed docs into pgvector + admin page to trigger re-index | 1 day |
| 2 | Anthropic SDK wiring + streaming endpoint + system prompt + chat UI | 2-3 days |
| 3 | Tool definitions (4 read-only tools) + tenant scoping + audit logging | 2 days |
| 4 | Per-tenant rate limit + spend cap UI + 80% alert | 1 day |
| 5 | Eval set (50-100 representative questions) + manual scoring | 1 day |
| 6 | Polish: tone, length tuning, panel UX, mobile | 1 day |

Total: ~8-10 dev days.

## Open questions

1. Do we want a chat UI per tenant, or per user-within-tenant? (Probably per user, but cross-tenant context is leak-prone.)
2. Inline citations (links back to help articles + audit events) — table-stakes or v3.2?
3. Voice input — operators have asked, but probably v3.2 noise.
4. Multi-turn memory beyond conversation buffer — anything stored long-term about user preferences? Probably no; defer.
